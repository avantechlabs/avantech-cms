# Impeccable Design Context — Avantech CMS Editor

> This file is the persistent design source of truth for the CMS **editor**.
> `frontend-design-impeccable` and related skills read this first.

## Reference handling (important)

`Working/w_2026_5_26/cms-flow.html` (in the coaching vault) is a **high-fidelity
mockup of the target feel** — not markup to transcribe. Follow the **design
language and interaction model that flow from it**, not its literal HTML/CSS.
Visuals are open to refinement; the interaction model is the locked part.

## Design Context

### Users

The primary editor is a **non-technical site owner** — e.g. the film studio
owner whose marketing site this CMS wraps. Their context: they want to change
words (and later images) on their **real, live site** without a developer, and
publish with confidence. The job-to-be-done is *"update my site and know exactly
what's live vs. what's just a draft."* An agency operator may use the same tool
across many client sites, but **the owner's clarity and safety always win** when
the two conflict. No fieldIds, schemas, or developer language ever surface in the
UI.

### Brand Personality

Calm · trustworthy · quietly premium. Voice is plain, reassuring, and never
technical. The editor should feel like a well-made instrument that gets out of
the way — closer to a designer tool's restraint than a SaaS admin dashboard's
density. Emotional goal: **calm, focused, in control; safe to experiment.**

### Aesthetic Direction

**Spatial model (locked):** site-on-a-stage. The customer's site renders on a
quiet canvas (it lives in an iframe). Editor chrome **frames** the canvas and
**recedes** until needed:
- a slim left **rail** for pages / collections / media,
- a floating **bottom action bar** (edit-status, Discard, Publish + change count),
- a right **side panel** for structured / contextual fields,
- a **View / Edit** mode toggle,
- and **in-place text editing** on the canvas with a light contextual toolbar
  (Notion-style selection toolbar) rather than a disconnected sidebar form.

**Visual language (starting point, open to refine):** derived from
`cms-flow.html` — gold `#FDB714` accent used sparingly for "live / act",
**Inter** for UI + **Space Mono** for labels/meta/eyebrows, **light + dark**
via CSS-variable tokens, soft shadows, pill controls, generous radius and quiet
space. Refine palette/type if a calmer, more premium canvas emerges (e.g. warmer
neutrals, even more restrained accent use). Color and emphasis are spent only on
what the editor must notice.

**Anti-references:** WordPress `wp-admin` and generic CRUD dashboards; loud SaaS
gradients; cluttered multi-row toolbars; anything that exposes schema/field
internals or feels like a "developer" tool.

### Design Principles

1. **Canvas over cockpit.** The live site is the hero on a quiet stage; chrome
   floats over it and appears on intent, not by default. Spatial like
   Framer/Webflow, but dialed all the way to calm.
2. **Draft vs. live is always legible.** A non-technical owner must never wonder
   what's published. Unpublished change is obvious, reversible, and only goes
   live on an explicit Publish.
3. **Edit in place.** Text changes where it lives, on the canvas, with a light
   contextual toolbar — never in a separate form disconnected from the result.
4. **Zero jargon, zero dead-ends.** Plain-language labels, every action has a
   safe undo/discard, nothing breaks by accident, no internal field names shown.
5. **The tool disappears.** Minimal chrome, restrained motion, generous quiet
   space. Polish is felt, not announced.
